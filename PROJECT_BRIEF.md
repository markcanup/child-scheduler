# PROJECT_BRIEF.md

## Goal

Build a cloud-hosted scheduling system to manage children’s wake-up times, bedtimes, and reminders.

- UI: Web application (hosted in AWS)
- Backend: AWS Lambda + DynamoDB
- Execution: Hubitat (local) via custom Groovy app
- No inbound connections to local network
- Hubitat pulls schedule and executes locally

---

## High-Level Architecture

### Flow

1. Hubitat → AWS
   - Push action catalog (available actions/devices)
   - Pull compiled schedule (next 7 days)

2. User → AWS UI
   - Configure schedules via web UI
   - Save configuration to backend
   - Backend compiles schedule

3. AWS → Hubitat
   - Hubitat periodically pulls compiled schedule
   - Executes events locally

---

## Key Design Principles

- Hubitat is execution engine only
- AWS is source of truth + scheduler
- No inbound access to home network
- Backend compiles schedule → Hubitat executes only
- Full-replace model for schedule config
- Broken references explicitly surfaced
- v1 scope is one user with one hub (single-tenant for now)

---

## Hubitat Capabilities (Validated)

- Custom app can call external HTTPS APIs
- Can schedule one-time executions locally
- Can invoke Rule Machine rules via RMUtils
- Can execute:
  - rule → Rule Machine
  - speech → device.speak(text)
  - notify → device.deviceNotification(text)
- Can expose:
  - selected Rule Machine rules
  - selected speech devices
  - selected notification devices

## Action Types (STRICT CONTRACT)

All actions must conform to the following schema:

### Common Fields

- actionType (string, required)
  - Allowed values: "rule", "speech", "notify"

- parameters (object, required)

---

### Rule Action

{
  "actionType": "rule",
  "parameters": {
    "targetId": "rule:<number>"
  }
}

Constraints:
- targetId must match regex: ^rule:\d+$
- rule must exist in action catalog
- no additional fields allowed

---

### Speech Action

{
  "actionType": "speech",
  "parameters": {
    "targetId": "speechTarget:<number>",
    "text": "string"
  }
}

Constraints:
- targetId must match: ^speechTarget:\d+$
- text:
  - required
  - max length: 500 characters
  - UTF-8 string
- device must support SpeechSynthesis capability

---

### Notify Action

{
  "actionType": "notify",
  "parameters": {
    "targetIds": ["notifyDevice:<number>", ...],
    "text": "string"
  }
}

Constraints:
- targetIds:
  - required array
  - min length: 1
  - max length: 10
  - each must match: ^notifyDevice:\d+$
- text:
  - required
  - max length: 500 characters
  - UTF-8 string
- each device must support Notification capability

---

### Broken Reference (OUTPUT ONLY)

{
  "eventId": "string",
  "validation": {
    "status": "broken",
    "message": "string",
    "originalLabel": "string"
  }
}

Constraints:
- never accepted from UI
- only generated by backend compiler

## Action Catalog

Owned by Hubitat.

Includes:
- actionDefinitions
- resources

Resource schema:

{
  "resourceId": "rule:<id> | speechTarget:<id> | notifyDevice:<id>",
  "label": "string",
  "type": "rule | speechTarget | notifyDevice"
}

Constraints:
- resourceId must be globally unique per hub
- label max length: 100 characters

---

## Schedule Model

### META

{
  "scheduleVersion": number,
  "compiledAt": "ISO-8601 timestamp",
  "timezone": "America/New_York"
}

Constraints:
- `timezone` is fixed to `America/New_York` for v1
- Future enhancement: Hubitat catalog updates may include hub timezone and update META
- `scheduleVersion` increments on every recompile (including recompiles triggered by catalog updates)

### Schedule Definition (DEF)

{
  "scheduleId": "string (UUID recommended)",
  "name": "string (max 100 chars)",
  "enabled": true,
  "daysOfWeek": ["MON", "TUE", ...],
  "timeMode": "absolute | relative",
  "baseTime": "HH:mm",
  "relativeToScheduleId": "string",
  "offsetMinutes": number,
  "actionType": "rule | speech | notify",
  "parameters": {...}
}

Constraints:
- scheduleId must be unique
- daysOfWeek must be valid enum values
- offsetMinutes range: -1440 to +1440
- baseTime required if timeMode=absolute
- relativeToScheduleId required if timeMode=relative

---

### Day Config (DAY)

{
  "date": "YYYY-MM-DD",
  "schoolDay": true,
  "notes": "string (optional, max 200 chars)",
  "overrides": {
    "scheduleId": {
      "enabled": true|false,
      "timeOverride": "HH:mm"
    }
  }
}

---

### Compiled Events (EVT)

{
  "eventId": "string",
  "date": "YYYY-MM-DD",
  "time": "HH:mm",
  "actionType": "...",
  "parameters": {...},
  "sourceScheduleId": "string"
}

---

### Broken References (BROKEN)

{
  "eventId": "string",
  "date": "YYYY-MM-DD",
  "message": "string",
  "originalLabel": "string",
  "sourceScheduleId": "string"
}

## API Endpoints

### Hubitat

POST /hubitat/action-catalog  
GET /hubitat/schedule  

Hubitat schedule query contract:
- `days` must be an integer between 1 and 90
- Day window is full-day based, beginning with current day in `META.timezone` as day 1
- Include all events for each included day, including already-past events from today

### UI

GET /catalog  
GET /schedule/config  
PUT /schedule/config  

---

## Compiler Responsibilities

- Compile next 7 days
- Resolve absolute and relative times
- Apply overrides
- Validate actions against catalog
- Detect broken references
- Output EVT and BROKEN items
- Recompile when schedule config is saved (`PUT /schedule/config`)
- Recompile when action catalog is updated (`POST /hubitat/action-catalog`)

---

## Authentication

- Hubitat: X-Hubitat-Token
- UI: Cognito JWT

Hubitat token operational policy (v1):
- PSK is managed via AWS deployment configuration (`HUBITAT_TOKEN`), not editable in UI
- Last-rotated timestamp is managed via `HUBITAT_TOKEN_LAST_ROTATED`
- PSK timestamp is visible in UI as read-only metadata
- Rotate annually
- UI should show a warning when key age reaches 365 days; do not hard-stop solely due to age

---

## Error Contract

Use a common error shape for validation and business-rule failures (including circular relative schedule references):

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Human-readable explanation"
  }
}

---

## Broken Reference Notification Policy

- Broken references should be surfaced once per `scheduleVersion`
- Broken-reference de-duplication key: `sourceScheduleId + date`
